### Engineer — 技术视角

**存储层事实：**
| 存储类型 | 内容 | 并发控制 |
|---------|------|---------|
| SQLite | User, AuthToken 表 | SQLAlchemy async session |
| JSON 文件 | workflows, sessions, kb_metadata, documents, processing_tasks | FileLock |
| ChromaDB | 向量索引 | @lru_cache 单例 |
| 文件系统 | 上传文档 | — |
- 数据迁移：无专用迁移脚本

**核心引擎事实：**
- 工作流引擎：BFS 遍历 + DFS 环检测，6 种节点，条件分支 via sourceHandle
- 变量解析：{{node_id.output}} 单次替换，simpleeval + JS→Python 语法转换
- 错误处理：节点错误→终止执行（无重试），状态不持久化
- RAG 管道：SentenceSplitter(512/50), SiliconFlow BGE-M3, ChromaDB, L2→相似度转换
- 技能执行：正则变量替换，必填校验，可选 RAG 集成

**API/协议事实：**
- SSE 事件：thought, token, citation, done, error（5 种）
- HTTP API：/auth, /workflows, /knowledge, /skills, /chat, /admin（6 个域）
- 超时：Token 7 天过期，任务保留 30 分钟，HTTP/LLM/SSE 无显式超时
- 客户端断连检测：无
- 背压控制：无

**安全机制事实：**
- 认证：Bearer Token（UUID v4），7 天过期，6 小时清理周期
- 输入验证：路径遍历校验、文件名安全化、文件类型/大小限制、Skill 名称/占位符校验、Pydantic Field 约束
- 沙箱/隔离：无代码沙箱（Skill 仅 prompt），simpleeval 禁用函数调用

**依赖和模块耦合事实：**
- 核心依赖：fastapi, sqlalchemy, chromadb, llama-index, openai, pydantic, filelock, simpleeval
- 模块导入：api → core → models，无 Protocol/ABC 抽象
- 单例模式：get_chroma_client(), get_rag_pipeline(), get_skill_executor(), get_client()

### 独到发现

#### PM 发现与 Engineer 交叉评论

**关于「RAG 知识管理部分实现——仅支持 .txt/.md，不支持 PDF/Word」**
[Engineer → PM] +1，理由：技术上需要引入新依赖（如 `pypdf` / `python-docx`），后端 `rag.py` 已有清晰的文档处理管道，扩展格式支持只需新增解析器函数。现有架构支持良好。[成本：4-6 小时]

**关于「竞品差距：多模型支持」**
[Engineer → PM] +1，但需澄清。技术上后端 `llm.py` 已封装 LLM 调用，改造成多模型只需抽象 LLM Provider 接口、新增配置项。DeepSeek 兼容 OpenAI SDK，改造成本低于预期。[成本：8-12 小时]

**关于「竞品差距：Docker 部署」**
[Engineer → PM] +1，理由：项目已有明确的依赖管理，Docker 化只需编写 Dockerfile + docker-compose.yml，技术阻碍为零。这是高 ROI 行动。[成本：2-4 小时]

**关于「竞品差距：可观测性（日志/监控）」**
[Engineer → PM] +1，理由：后端已有结构化日志可扩展，引入 OpenTelemetry 或简单请求追踪器成本可控。但需明确监控目标。[成本：8-16 小时，取决于监控深度]

**关于「竞品差距：团队协作/多租户」**
[Engineer → PM] 反驳，理由：这是架构级改造，需引入 Organization/Team/Membership 模型、重构权限系统、所有资源加 tenant_id 隔离。与「轻量级」定位冲突，建议不纳入核心路线。[成本：40-80 小时]

**关于「竞品差距：国际化（i18n）」**
[Engineer → PM] +1，理由：Vue 生态有 `vue-i18n`，前端改造成本中等。后端 API 返回消息、Skill 提示词也需国际化。[成本：前端 12-16h + 后端 8-12h]

#### Designer 发现与 Engineer 交叉评论

**关于「响应式设计不足——仅 4 处 @media 查询，存在固定宽度值」**
[Engineer → Designer] +1，理由：CSS 改造无阻碍。但需确认业务目标——项目定位「桌面开发者工具」，移动端适配优先级应低于核心功能。[成本：8-12 小时]

**关于「无障碍不足——仅 9 处 aria 属性，label 关联不完整」**
[Engineer → Designer] +1，理由：aria 属性添加成本极低。但需权衡 ROI——目标用户是技术人员，屏幕阅读器使用率可能较低。[成本：4-6 小时]

**关于「组件不一致——Button vs 原生 button 混用（30 vs 37）」**
[Engineer → Designer] +1，理由：统一为 Button 组件无阻碍。需确认原生 button 是否有特殊场景（如原生 form submit）。[成本：2-3 小时]

#### Engineer 交叉发现

**1. 无客户端断连检测与背压控制**
SSE 流式响应在客户端断开后无感知，浪费 LLM API 调用额度。应实现心跳或客户端主动取消检测。[成本：4-6 小时]

**2. 工作流执行错误无重试机制**
节点执行失败直接终止，无重试或降级策略。对于调用外部 LLM API 的节点，瞬态失败会导致整个工作流失败。[成本：6-8 小时]

**3. 数据无迁移机制**
SQLite + JSON 文件 + ChromaDB 三种存储均无版本化迁移脚本。Schema 变更后需手动处理数据兼容，生产环境风险高。[成本：8-12 小时]

**4. simpleeval 表达式能力受限**
变量解析已禁用函数调用，表达式能力受限（如无法做字符串 contains，需 workaround）。长期应考虑更安全的表达式引擎或预定义函数白名单。[成本：4-8 小时]

**5. 测试覆盖率未量化**
项目有 11 个测试文件约 45 个测试，但无覆盖率报告。建议引入 `pytest-cov` 和 Vitest coverage。[成本：2-3 小时]

## Round 3：共识层

### ✅ 共识项

1. **议题：SSE 断连检测缺失**
   - 结论：三方向意这是生产可用性基础门槛，影响 API 成本和用户体验。
   - 建议行动：实现客户端断连检测，在 API 层添加 `Request.disconnected()` 检测 + 生成器提前终止。
   - 成本估算：4 小时
   - 优先级：P0

2. **议题：Docker 部署缺失**
   - 结论：与「开发者工具」定位矛盾，是开发者第一部署需求。
   - 建议行动：编写多阶段构建 Dockerfile + docker-compose.yml + 更新 README。
   - 成本估算：3 小时
   - 优先级：P0

3. **议题：loading 状态完善**
   - 结论：知识库处理、工作流执行、技能调用等异步操作缺少前端 loading 指示器。
   - 建议行动：复用现有 Button 组件 loading 状态，或添加全局 loading overlay。
   - 成本估算：6 小时
   - 优先级：P0

4. **议题：RAG 文件格式扩展（PDF/Word）**
   - 结论：直接影响用户获取，开发者知识库来源往往包含 PDF/Word。
   - 建议行动：引入 PyMuPDF/python-docx 解析库，更新 API 校验。
   - 成本估算：8 小时
   - 优先级：P1

5. **议题：工作流错误恢复**
   - 结论：长时间工作流因瞬态失败中断，直接影响用户留存。
   - 建议行动：先实现执行状态持久化（断点续跑），再增加节点级重试机制。
   - 成本估算：16 小时（持久化 8h + 重试 8h）
   - 优先级：P1

6. **议题：可观测性缺失**
   - 结论：生产就绪性缺口，无结构化日志、无 tracing、无 metrics。
   - 建议行动：集成 OpenTelemetry + 结构化 JSON 日志 + Prometheus metrics endpoint。
   - 成本估算：16 小时
   - 优先级：P1

7. **议题：表单 label 关联修复**
   - 结论：影响所有用户的可用性，label 点击聚焦是通用交互期望。
   - 建议行动：逐个视图添加 label/id 关联。
   - 成本估算：4 小时
   - 优先级：P2

8. **议题：Skill 变量校验**
   - 结论：用户提供错误变量名时产生无效 prompt。
   - 建议行动：替换后检测剩余 `{{...}}` 并发出警告事件。
   - 成本估算：2 小时
   - 优先级：P1

### ⚡ 分歧项

1. **议题：无障碍支持优先级**
   - 各方立场：
     - PM：P2，目标用户群体无障碍需求占比低，不影响用户获取
     - Designer：P1，开发者中也有障碍用户，且影响企业形象
     - Engineer：P1，aria 属性添加成本极低
   - 裁决/建议：作为 P2，但在其他 P0/P1 行动完成后应优先处理表单 label 关联（已纳入共识项）。
   - 决策依据：PM 的「不影响用户获取」论据成立，但 Engineer 的「成本极低」论据也成立。折中为 P2。

2. **议题：响应式设计增强**
   - 各方立场：

     - Designer：需要，平板/小笔记本用户体验差
     - Engineer：需要，但优先级应低于核心功能
   - 裁决/建议：保持现状，作为远期规划。当前定位明确为「桌面开发者工具」。
   - 决策依据：PM 的「移动端非目标场景」论据成立，Vue Flow 库对移动端支持也有限。

3. **议题：多租户/团队协作**
   - 各方立场：

     - Designer：未表态
     - Engineer：反驳，与「轻量级」定位冲突，成本 80+ 小时
   - 裁决/建议：不纳入当前路线图。在 README 中明确说明「当前版本聚焦个人开发者场景」。
   - 决策依据：Engineer 的「与定位冲突」和「成本 80+ 小时」论据成立。
