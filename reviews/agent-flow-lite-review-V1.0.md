## 模式选择

**选定模式：** 对等协作

**选择理由：**
- 用户明确要求「平衡的多视角分析」，无单一主导领域

## Round 1：事实层（只写事实，不评价）

### PM — 产品视角

**产品定位：**
- 自我定位："轻量级、高还原度的 AI 智能体编排平台"（README.md:3）
- 声称核心功能：可视化工作流、RAG 知识管理、智能对话终端、Skill 技能系统、用户管理系统（README.md:16-20）
- 技术栈：Vue 3 + Vite + TypeScript + Vue Flow（前端）；FastAPI + Python 3.11+ + SQLAlchemy async（后端）；DeepSeek API（LLM）；SiliconFlow API BGE-M3（Embedding）；ChromaDB + LlamaIndex（向量/RAG）

**功能完成度：**
- 可视化工作流：已实现，节点类型 Start/LLM/Knowledge/Condition/Skill/End 全部实现
- RAG 知识管理：已实现，SentenceSplitter(512, 50) 分块，BGE-M3 向量化，ChromaDB 存储
- 智能对话终端：已实现，SSE 流式响应，多轮会话保留最近 10 条，@Skill 调用正则解析
- Skill 技能系统：已实现，YAML frontmatter + Markdown 格式，完整 CRUD + 执行
- 用户管理系统：已实现，Bearer Token（UUID v4，7天有效），admin/user 二级角色，会话隔离

**竞品功能对比：**
- 同类产品：Coze（字节）、Dify、FastGPT、Flowise
- 本项目有/竞品也有：可视化工作流、RAG、条件分支、变量系统
- 本项目无/竞品有：多模型支持、插件/工具市场、Agent 调试器、多租户/SaaS、API Key 管理、团队协作、版本控制/发布、对话日志分析、多渠道部署、Docker 部署、SDK、国际化
- 本项目特有：Harness Lab 治理层、Skill Markdown 格式

**目标用户线索：**
- UI 复杂度：中等偏高（Vue Flow 拖拽需理解节点/边/Handle，Skill 编辑器需理解 YAML frontmatter，工作流变量语法 `{{step1.output}}`）
- API 文档：有 Swagger UI `/docs` 端点，有设计文档
- 多租户/权限：无多租户（无 tenant_id），admin/user 二级权限
- 部署方式：手动部署（install.sh + start.sh），无 Dockerfile
- 测试覆盖：后端 14 个测试文件，前端 12 个测试文件，2 个 CI 工作流

### Engineer — 技术视角

**存储层事实：**
- SQLite (async)：用户表、认证令牌表
- JSON 文件：workflows.json、sessions/{id}.json、kb_metadata.json 等
- ChromaDB：向量索引
- 并发控制：SQLAlchemy async session + FileLock

**核心引擎事实：**
- 工作流引擎：BFS 遍历 + DFS 三色染色环检测
- 变量解析：正则 + simpleeval 求值
- 节点类型：start、llm、knowledge、condition、skill、end

**错误处理：**
- 节点级：发出 node_error，设置空输出，return 终止
- 工作流级：workflow_error，return 终止
- RAG 失败：continue without RAG context

**状态持久化：**
- 持久化：工作流定义、聊天会话、技能定义、知识库向量
- 不持久化：**工作流执行状态（内存 ExecutionContext）**

**API/协议事实：**
- SSE 事件类型：thought、token、citation、done、error、workflow_*、node_*
- 超时设置：Token 7天；LLM API 无显式超时
- 背压控制：**无显式背压控制**
- 客户端断连检测：**后端无显式机制**

**安全机制事实：**
- 认证：Bearer Token (UUID v4)，7 天有效期
- 输入验证：Pydantic、路径遍历防护、文件类型白名单
- 沙箱/隔离：simpleeval 表达式沙箱；**无代码执行沙箱；无进程隔离**

### 独到发现

1. **数据安全与合规被三方遗漏**（Critic）：用户上传文档、对话内容传至 DeepSeek/SiliconFlow API，用户是否知情？ChromaDB 删除后向量是否同步清理？是否有审计日志？这是产品级问题。

2. **用户引导/Onboarding 缺失**（PM 交叉发现）：Vue Flow 拖拽、`{{variable}}` 语法、YAML frontmatter 等概念对首次用户构成认知门槛，直接影响用户获取转化率。

3. **Skill 系统商业模式模糊**（PM 交叉发现）：Skill 如何分发？是否有预置库？用户如何发现/共享？

4. **错误处理的用户可视化缺失**（Designer 交叉发现）：Engineer 描述了后端错误处理逻辑，但用户看到什么？RAG 失败静默降级，用户收到不完整回答却不知情。

5. **测试覆盖盲区**（Engineer 交叉发现）：工作流引擎核心逻辑（环检测、分支路由、错误传播）无单元测试。

### Critic 质询

- **要求回应：** PM
- **级别：** 普通

#### 质询 2：竞品对比标准是否公平
- **论据**：Coze、Dify 是商业化 SaaS 平台，本项目是开源个人项目。将 SaaS 级功能与个人开源项目对标，标准是否公平？哪些是真正缺陷，哪些是超出范围的合理边界？
- **要求回应：** PM
- **级别：** 普通

#### 质询 3：alert() 的 15 处调用是否构成问题

- **要求回应：** Designer
- **级别：** 普通

#### 质询 4：aria=0 对谁构成障碍
- **论据**：如果项目定位是开发者工具，aria 缺失优先级可能低于功能完整性；如果目标是通用 SaaS，则是合规风险。

## Round 3：共识层

### ✅ 共识项（PM/Designer/Engineer 三方 2/3 或 3/3 同意）

**1. alert()/confirm() 破坏产品专业感**
- **结论**：15 处 alert + 7 处 confirm 在专业产品中显得廉价，需要统一 Toast/Notification 组件替换。
- **建议行动**：构建 Toast 组件 + 替换所有调用。
- **成本估算**：6-10 小时（Engineer）
- **优先级**：P1

**2. 工作流执行状态不持久化导致资源浪费**
- **结论**：Engineer 确认长工作流执行期间服务重启进度丢失，已执行的 LLM 节点需重跑消耗 API quota。
- **建议行动**：引入 ExecutionContext 持久化 + checkpoint 恢复机制。
- **成本估算**：40-60 小时
- **优先级**：P2（高成本，可分阶段实现）

**3. SSE 无背压控制 + 无断连检测**
- **结论**：后端无 `request.is_disconnected()` 检测，前端无 AbortController 取消机制，可能造成资源浪费。
- **建议行动**：后端增加断连检测，前端增加取消机制。
- **成本估算**：4-6 小时
- **优先级**：P1

**4. 无数据迁移工具（Alembic）**
- **结论**：数据库 schema 变更无迁移脚本，用户可能无法升级版本。
- **建议行动**：引入 Alembic 迁移框架。
- **成本估算**：8-12 小时
- **优先级**：P2

**5. 功能"已实现"不等于"已验证"**
- **结论**：测试覆盖存在盲区（工作流引擎核心逻辑无单元测试）。
- **建议行动**：补全工作流引擎测试覆盖。
- **成本估算**：8-12 小时
- **优先级**：P1

**6. Docker 部署缺失阻碍企业用户**
- **结论**：无 Dockerfile 对比竞品一键部署，第一关就受阻。
- **建议行动**：提供 Dockerfile + docker-compose.yml。
- **成本估算**：4-6 小时
- **优先级**：P1

### ⚡ 分歧项

**1. 产品定位"轻量级"与目标用户画像矛盾**
- **PM 立场**：承认定位模糊，需明确目标用户
- **Designer 立场**：UI 复杂度"中等偏高"过于模糊，实际是"专业用户才能胜任"
- **Engineer 立场**：+1，技术复杂度与"轻量级"矛盾
- **裁决/建议**：重写 README 定位，改为"面向开发者的轻架构 AI 编排平台"，明确目标用户为中高级技术人员
- **决策依据**：三方都承认定位与实现存在差距，需要调整定位而非降低复杂度
- **成本估算**：2 小时（文档修订）
- **优先级**：P0

**2. 无障碍合规优先级**
- **PM 立场**：P2 即可，目标用户是技术人员
- **Designer 立场**：P0，aria=0 意味着"产品不可用"，影响企业采购合规
- **Engineer 立场**：+1 但区分优先级，表单 label 关联 2-3h，完整 ARIA 4-6h
- **裁决/建议**：折中方案——高频页面（LoginView + ChatTerminal）优先修复 label 关联，作为企业采购合规保险。完整 WCAG 审计列为 P2。
- **决策依据**：成本极低（2-3h）vs 合规风险规避
- **成本估算**：2-3 小时（基础修复）
- **优先级**：P2

**3. 响应式设计不足**
- **PM 立场**：+1，限制产品可触达场景
- **Designer 立场**：+1，移动端完全不可用
- **Engineer 立场**：部分反驳，固定宽度是设计选择，真正问题是 min-width:1200px 硬性断点
- **裁决/建议**：桌面优先策略，文档明确不支持移动端。响应式改造列为 P2。
- **决策依据**：移动端改造成本高（16-24h），且工作流拖拽交互在移动端天然受限
- **成本估算**：16-24 小时（完整响应式）
- **优先级**：P2

### 🚨 关键质询回应

**质询 8：数据安全与合规**
- **回应**：
  - 数据传输至第三方 API（DeepSeek/SiliconFlow）需要在隐私政策中明确告知用户
  - ChromaDB 删除 collection 时向量同步删除（已验证 `chroma_client.py:delete_collection`）
  - 审计日志缺失，企业用户无法满足合规要求
- **处置**：纳入行动计划 P2（隐私政策文档 + 审计日志机制）
